# V1-1-22 — 운영사(SweatBridge Team) 전용 어드민 페이지 구축

## 1. 배경 및 목적

현재 스웻브릿지 웹 플랫폼은 개별 짐(박스) 단위의 코치들이 소속 회원을 관리하고 운영하는 데 최적화되어 있다. 하지만 서비스가 확장됨에 따라, **전체 고객사(박스)를 한눈에 파악하고 온보딩하거나 설정을 관리해야 하는 운영사(SweatBridge Team)를 위한 통합 관리 도구**가 부재한 상태다.

이를 해결하기 위해 일반 코치 계정과는 분리된 운영사 전용 어드민 페이지를 구축하여, 고객사 관리 효율성을 높이고 서비스 운영을 체계화한다.

---

## 2. 현재 상태 분석

### 2-1. 인증 / 권한 구조
- `User` 타입 (`src/types/auth.ts`)에 `role: string` 필드는 이미 존재.
- 그러나 `ProtectedRoute` (`src/components/ProtectedRoute.tsx`)는 `isAuthenticated`만 검사하고 `role` 분기는 하지 않는다.
- `AuthContext`도 `role`을 별도로 노출하거나 라우팅 권한 체크에 활용하지 않고 있다.

### 2-2. 데이터 구조
- 모든 비즈니스 데이터는 `/box/{boxName}/...` 하위로 격리되어 있다.
- `boxName`은 로그인 시 `localStorage`에 저장되어 모든 Repository가 그 값을 참조한다.
- 운영사 어드민은 **특정 boxName에 종속되지 않고 `/box` 컬렉션 전체**를 횡단 조회해야 한다 → 별도의 Service/Repository 계층이 필요.

### 2-3. 라우팅 / 레이아웃
- `App.tsx`의 모든 보호된 라우트는 `MainLayout` (코치용 사이드바) 안에서 동작한다.
- 어드민은 다른 사이드바 메뉴 / 다른 헤더 컨텍스트가 필요하므로 별도 레이아웃을 둔다.

## 3. 작업 범위

### 3-1. 인증 및 권한 (Phase 1)
- [ ] `User.role` 값을 정식 enum으로 정의: `coach` | `operator` | `admin` (`src/types/auth.ts`)
- [ ] `AuthContext`에 `role` getter 및 `isOperator`, `isAdmin` 헬퍼 추가
- [ ] `ProtectedRoute`를 확장해 `requiredRoles?: UserRole[]` prop 지원
- [ ] 운영사 전용 라우트 가드 컴포넌트 `AdminProtectedRoute` 도입 (operator/admin만 통과, 그 외 `/dashboard`로 리다이렉트)

  - `/admin/dashboard` — 통합 지표 (Phase 3에서 활성화)

## 5. 영향 범위 / 회귀 리스크

- `User.role`을 enum 화 → 기존에 role 문자열을 직접 비교하던 곳이 있는지 grep 필요 (현재 코드에는 거의 없음).
- `ProtectedRoute` 시그니처 변경 시 `App.tsx`의 모든 `<ProtectedRoute />` 사용처 점검.
- 신규 박스 온보딩 시 빈 하위 컬렉션이 다른 코치 화면에서 어떤 fallback을 보이는지 확인 필요 (예: 빈 `lockers/lockerdoc`).
- `AdminBoxRepository.listAllBoxes()`는 `/box` 컬렉션 전체 권한이 필요 → **Firestore 보안 규칙에서 operator/admin role에 한해 read 허용** 필수 (백엔드 작업 동반).

---

## 6. 사전 결정 필요 항목

1. **신규 박스 온보딩 시 코치 계정 생성 정책**
   - 박스만 만들고 코치 계정 생성은 별도 흐름인지, 동시에 처리하는지.
2. **운영 상태(`status`) 비즈니스 정의**
   - `suspended` 박스의 코치 로그인 차단 여부.
3. **AdminDashboard 표시 지표 우선순위**
   - 전체 매출 합계 / 박스별 회원수 / 신규 가입 트렌드 중 무엇부터?
4. **Firestore 보안 규칙 작성 주체**
   - 본 작업에 포함할지, 별도 백엔드 티켓으로 분리할지.
