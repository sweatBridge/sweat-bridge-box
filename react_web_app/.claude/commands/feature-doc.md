usage: /feature-doc [기능명 또는 주제]

## 역할

사용자가 지정한 기능 또는 주제에 대해 `docs/features/{기능명}.md` 문서를 생성하거나 업데이트한다.
이 문서는 **다른 서비스나 모듈과 공유하기 위한 것**이므로, 코드 구현 상세는 제외하고 **개념 · 규칙 · 흐름** 중심으로 작성한다.

---

## 실행 절차

### 1. 기능명 확인

- `/feature-doc` 뒤에 인자가 있으면 그것을 기능명으로 사용한다.
- 인자가 없으면 사용자에게 "어떤 기능을 문서화할까요?" 라고 묻고 답변을 기다린다.

### 2. 기존 문서 확인

- `docs/features/` 디렉토리를 확인한다.
- 동일한 이름(또는 유사한 이름)의 파일이 이미 있으면 사용자에게 알리고, 덮어쓸지 추가할지 묻는다.

### 3. 코드베이스 탐색 (필요한 경우)

- 사용자가 이미 충분한 설명을 제공했다면 탐색을 생략해도 된다.
- 부족하다면 관련 파일(controller, service, repository, model, view)을 읽어 개념을 파악한다.
- **탐색 목적**: 코드를 설명하기 위해서가 아니라, 그 안에 담긴 **비즈니스 규칙과 데이터 흐름**을 추출하기 위해서다.

### 4. 문서 작성

아래 템플릿을 기준으로 작성한다. 모든 섹션이 필요하지 않을 수 있으니 내용에 맞게 조정한다.

```markdown
# {기능명}

## 개요
이 기능이 무엇을 하는지, 왜 존재하는지 2~4문장으로 설명한다.

## 핵심 개념
기능을 이해하는 데 필요한 용어와 개념을 정의한다.

## 규칙 및 제약
- 이 기능이 따르는 비즈니스 규칙
- 허용/불허 조건
- 예외 처리 방식

## 상태 및 전환
상태가 있는 기능이라면 상태 다이어그램이나 표로 전환 흐름을 표현한다.

## 주요 흐름
사용자 또는 시스템 관점에서 기능이 어떻게 동작하는지 단계별로 설명한다.
시나리오가 여럿이면 각각 나누어 기술한다.

## 데이터 구조 (개념)
코드가 아닌 개념 수준에서 어떤 데이터가 관리되는지 설명한다.
필드명과 의미, 관계를 서술한다.

## 다른 기능과의 관계
이 기능이 의존하거나 영향을 주는 다른 기능을 나열한다.
```

### 5. 파일 저장

- 경로: `docs/features/{기능명}.md`
- 파일명은 소문자 + 하이픈 형식으로 정규화한다. (예: `box-membership`, `class-reservation`)

### 6. README 업데이트

- `docs/README.md`에 새 문서 링크가 없으면 목록에 추가한다.

---

## 작성 원칙

- **코드 없음**: 함수명, 클래스명, 파일 경로는 포함하지 않는다.
- **구현 독립성**: 이 문서를 읽는 사람이 Flutter나 Firebase를 몰라도 이해할 수 있어야 한다.
- **명확한 규칙**: "~할 수 있다"보다 "~해야 한다 / ~할 수 없다"처럼 단정적으로 서술한다.
- **간결함**: 섹션당 필요한 내용만 담고, 불필요한 섹션은 생략한다.
